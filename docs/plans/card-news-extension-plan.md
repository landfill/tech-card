# 카드뉴스 확장 계획 (보완)

기준: [cakepower 카드뉴스 스킬 정리](https://www.cakepower.net/post/%EC%B9%B4%EB%93%9C-%EB%89%B4%EC%8A%A4%EB%A5%BC-%EB%B0%9C%ED%96%89%ED%95%98%EB%8A%94-skill-%EC%A0%95%EB%A6%AC%ED%95%A9%EB%8B%88%EB%8B%A4-0c03e1/)  
현재 코드베이스: 데일리 인텔리전스 뉴스레터 (일단위).

---

## 1. 주단위 vs 일단위 · "10장"의 유래

- **레퍼런스**: 주단위 뉴스레터였다. 한 주에 여러 테마(표지, 하이라이트, 뉴스 5개, 인사이트 2개, 마무리)를 담기 위해 **카드 10장** 구조를 쓰고, **카드마다 다른 배경 이미지**(10장)를 생성했다.
- **현재 코드베이스**: 일단위 레터 — 하루에 **한 테마/한 편**이다.
- **적용 방식**: **데일리 카드 뉴스**로, 한 호(하루)당 **슬라이드(카드) 여러 장**은 그대로 둘 수 있으나, 레터 주제가 하루에 하나이므로 **배경 이미지는 1호당 1장**만 생성한다.
  - "10"이라는 숫자는 레퍼런스의 **주간용 카드 개수**에서 온 것이며, 데일리 레터에서는 **배경 10장이 필요하지 않다**. 데일리 뉴스레터 1개당 **배경 이미지 1개**면 충분하다.
  - 카드 슬라이드 수(예: 5~10장)는 UI/본문 분할용으로 유지할 수 있고, **모든 슬라이드가 그날 생성한 배경 1장을 공유**한다.

---

## 2. 배경 이미지: 나노바나나 API — 1호당 1장

- **나노바나나(nanobanana) API**로 **레터 1호당 배경 이미지 1장**만 생성한다.
- 흐름:
  1. 해당 호 레터 마크다운(헤드라인·오프닝·섹션 요약)에서 **그날의 테마/분위기**를 나타내는 이미지 컨셉 1개 추출.
  2. nanobanana용 JSON 프롬프트(또는 API 스펙에 맞는 입력) 1개 생성.
  3. API 호출 1회로 이미지 1장 생성 후 저장 (예: `frontend/public/images/cards/YYYYMMDD.png` 또는 `YYYYMMDD_bg.png`).
- 파이프라인: `card_generate` 다음 **`card_backgrounds`** 스텝에서 나노바나나 연동으로 **해당 날짜당 1회만** 호출. (실패 시 해당 호 카드뉴스는 그라디언트 폴백.)

## 3. 프론트엔드 UI 변경 (필수)

현재 프론트엔드([frontend/src/App.tsx](frontend/src/App.tsx))는 다음만 있다.

- **탭**: "발송 내역" | "파이프라인"
- **발송 내역**: 날짜 목록 → 선택 시 해당 날짜 레터 **본문만** ReactMarkdown으로 표시, 피드백/삭제

카드뉴스를 사용하려면 **UI 변경이 필요**하다. 제안하는 변경 사항은 아래와 같다.

### 3.1 진입점 (택일 또는 병행)

- **옵션 A — 레터 상세 내 전환**  
  - 레터 날짜 선택 후 본문 보기 화면에 **"카드로 보기"** 버튼 추가.  
  - 클릭 시 같은 날짜의 카드 데이터(`GET /api/letters/{date}/cards`)를 불러와 **카드뉴스 전용 뷰**로 전환.  
  - "본문 보기" 버튼으로 다시 마크다운 본문 뷰로 복귀.
- **옵션 B — 상단 탭 추가**  
  - 현재 "발송 내역" | "파이프라인" 옆에 **"카드뉴스"** 탭 추가.  
  - "카드뉴스" 탭: 날짜 목록(카드가 생성된 날짜만 또는 전체 중 카드 있는 날 강조) → 선택 시 해당 날짜 카드뉴스 뷰만 표시.

**권장**: 옵션 A를 기본으로 하고, 필요 시 옵션 B를 추가해 "카드뉴스만 보고 싶을 때" 직접 진입할 수 있게 함.

### 3.2 새로 추가할 UI 요소

| 구분 | 내용 |
|------|------|
| **CardNews 컴포넌트** | 9:16 세로형, **해당 호 공통 배경 1장**(나노바나나 1호당 1장)을 모든 슬라이드에서 공유, 또는 그라디언트 폴백. 진행 바, 좌우 스와이프·키보드 네비게이션. 레퍼런스 CardNews.tsx 디자인 시스템 참고. |
| **카드 뷰 전용 화면** | 위 컴포넌트로 `cards` JSON 렌더링. 상단에 날짜·"본문 보기" 등 안내. |
| **API 연동** | `GET /api/letters/{date}/cards` (또는 `GET /api/cards/{date}`)로 카드 JSON + 이미지 경로 사용. |

### 3.3 상태·라우팅

- 카드 데이터가 없는 날짜: "카드로 보기" 비활성화 또는 "이 호는 카드뉴스가 없습니다" 메시지.
- 카드 뷰에서 사용하는 배경 이미지 경로는 호당 1장 기준으로 통일 (예: `public/images/cards/YYYYMMDD.png`).

## 4. 파이프라인 구조 · 기존 파이프라인 변경 여부

### 4.1 실행 구조: 하나의 파이프라인, 뒤에 스텝 추가

- **별도 “카드뉴스 파이프라인”이 있는 것이 아니다.** 기존 파이프라인(collect → … → letter_generate) **뒤에** 카드 관련 스텝이 **같은 파이프라인에** 붙는 형태다.
- 한 번 `run_pipeline(date)`를 호출하면: 기존 5단계 실행 후 **이어서** `card_generate` → `card_backgrounds`가 순서대로 실행된다. 스케줄러·API는 기존처럼 `run_pipeline`만 호출하면 된다.

```
collect → analyze → summarize → dedup → letter_generate → card_generate → card_backgrounds
```

### 4.2 기존 파이프라인 코드 변경점 (있음)

- **기존 스텝 로직(collect, analyze, summarize, dedup, letter_generate)**: **변경 없음.** 입력·출력·동작 그대로 둔다.
- **변경이 필요한 부분**:
  - **`pipeline/runner.py`**: `PIPELINE_STEPS`에 `"card_generate"`, `"card_backgrounds"` 추가. `run_step()`에 두 스텝 분기 추가. `run_pipeline()` 반환값은 기존 유지(letter_path 등) 또는 카드 결과를 보조로 포함할 수 있음.
  - **`backend/routers/pipeline.py`**: `STEP_NAMES`에 `"card_generate"`, `"card_backgrounds"` 한글 표기 추가 (UI/상태 표시용).
  - (선택) `pipeline/__main__.py`: 완료 메시지에 카드 경로 등 출력 추가.

즉, **기존 파이프라인 “코드”는 수정이 필요**하고(스텝 목록·분기·라우터 표시), **기존 스텝 “동작”은 그대로**다.

### 4.3 기존 생성 레터로 카드 파이프만 테스트

- **가능하다.** 이미 생성된 데일리 레터 파일(`data/letters/YYYY-MM-DD.md`)만 있으면, **카드 단계만** 돌려서 카드뉴스를 만들 수 있다.
- **방법**: `run_pipeline(date, ..., from_step="card_generate")` 호출. 기존 코드에 이미 `from_step`이 있으므로, `PIPELINE_STEPS`에 카드 스텝을 넣은 뒤에는 `from_step="card_generate"`로 해당 날짜부터 끝까지(card_generate → card_backgrounds)만 실행된다. collect ~ letter_generate는 실행되지 않는다.
- **전제**: `card_generate`는 **레터 내용을 체크포인트가 아니라 디스크의 레터 파일**에서 읽도록 구현한다. 즉 `letter_path(data_dir, date)`로 `data/letters/{date}.md`를 읽어 입력으로 사용. 그래야 이미 만들어 둔 레터만으로 카드 단계를 독립적으로 테스트할 수 있다.
- **예**: `data/letters/2026-03-16.md` 등 기존 레터가 있는 날짜를 골라, CLI 또는 API에서 `from_step=card_generate`로 파이프라인 실행 → 해당 호에 대해 카드 JSON·배경 이미지만 생성되는지 확인.

## 5. 파이프라인·백엔드 요약

- **스텝 순서**: … → `letter_generate` → **`card_generate`** → **`card_backgrounds`** (나노바나나 **1호당 1장** 생성) → (선택) publish.
- **card_generate**: 레터 마크다운 → LLM으로 카드 슬라이드 JSON 생성(장수는 5~10 등 설계에 따름) → `data/cards/YYYY-MM-DD.json` 저장.
- **card_backgrounds**: 레터에서 그날 테마 1개 추출 → 나노바나나 API **1회** 호출 → 이미지 1장 저장 (예: `YYYYMMDD.png`), JSON에 `bgImage`(호당 공통 경로) 보강.
- **백엔드**: `GET /api/letters/{date}/cards` (및 필요 시 카드 목록/존재 여부 API) 추가.
- **프론트**: 위 3장 대로 UI 변경 적용.

## 6. 구현 후 검증 및 레포팅

구현이 끝난 뒤 **셀프 검증**을 수행하고, 결과를 **레포팅**한다.

- **검증 절차** (구현 주체가 직접 수행):
  - 기존 레터가 있는 날짜(예: `data/letters/` 내 한 날짜)에 대해 `from_step=card_generate`로 파이프라인 실행 → card_generate·card_backgrounds 정상 완료 여부 확인.
  - 생성물 확인: `data/cards/YYYY-MM-DD.json` 존재·스키마·내용, 배경 이미지 1장(예: `public/images/cards/YYYYMMDD.png`) 존재 여부.
  - API 확인: `GET /api/letters/{date}/cards` 호출 시 위 JSON 및 이미지 경로 반환 여부.
  - 프론트 확인: 레터 상세에서 "카드로 보기" 진입 → 카드 뷰·스와이프·배경 표시 정상 여부.
  - (선택) 기존 파이프라인 전체(collect부터) 한 날짜로 실행 → 카드 단계까지 포함 정상 종료 여부.
- **레포팅**: 검증 결과를 문서로 정리해 레포에 반영한다. 예: `docs/card-news-verification-YYYYMMDD.md` 또는 기존 `docs/newsletter-*-verification-report.md` 형식을 참고한 검증 리포트. 포함할 내용: 검증 일시, 사용한 날짜/데이터, 검증 항목별 통과 여부, 이슈·한계점 요약.

## 7. 이메일 발송 (SMTP·발송 수단)

레터 본문·카드뉴스를 **이메일로 발송**할 때, 앱은 **SMTP 클라이언트**로 기존 메일 서버(또는 외부 발송 서비스)에 접속해 보낸다. 수신자 목록은 `data/subscribers.json` 등에 두고, **발송을 수행할 서버/계정**은 설정으로 지정한다.

### 7.1 회사 SMTP 사용 시 (정책 확인)

- **회사 메일 주소를 갖고 있다고 해서** 회사 SMTP를 스크립트/앱에서 쓸 수 있는지는 **회사 정책에 따라 다르다.**
- 많은 회사는 보안상 **Outlook/웹메일 등 허용 클라이언트만** 발송 가능하게 하고, **앱·스크립트의 SMTP 접속은 차단**하거나 OAuth/전용 API만 허용한다.
- **결론**: 회사 SMTP 사용 전 **반드시 내부 정책·IT 부서 확인**이 필요하다. 불확실하면 아래 Gmail 또는 SendGrid/메일건 등 **발송 전용 서비스**를 사용하는 편이 안전하다.

### 7.2 Gmail을 통한 발송

- **호스트**: `smtp.gmail.com`  
- **포트**: `587` (TLS 권장) 또는 `465` (SSL)  
- **인증**: 2단계 인증 사용 시 **앱 비밀번호** 사용 (일반 로그인 비밀번호는 사용 불가). [Google 계정 → 보안 → 앱 비밀번호](https://myaccount.google.com/apppasswords)에서 생성.
- **발송 한도**: 일반 Gmail 약 **500통/일**, Google Workspace 약 **2,000통/일**. 구독자 수가 많으면 한도 초과에 유의.
- **용도**: 회사 SMTP가 불확실할 때 **테스트·개인용**으로 사용하고, 필요 시 나중에 회사 SMTP 또는 SendGrid 등으로 설정만 교체하면 된다.

### 7.3 설정 요약

- `config/email.yaml`(또는 env)에 `smtp_host`, `smtp_port`, `from_addr` 등 저장.
- `smtp_user`, `smtp_password`는 **비밀번호 보안**을 위해 **환경 변수(.env)** 에서만 로드하는 것을 권장.

## 8. 정리

- **주단위 vs 일단위**: 데일리 레터 → **데일리 카드 뉴스**. "10장"은 레퍼런스(주간)의 카드 개수에서 온 것이며, **배경은 1호당 1장**이면 충분하다.
- **배경 이미지**: **나노바나나 API**로 **레터 1호당 1장**만 생성. 모든 카드 슬라이드가 이 배경을 공유.
- **프론트엔드**: **UI 변경 필요**. 레터 상세에 "카드로 보기" + 카드 전용 뷰(CardNews 컴포넌트), 필요 시 "카드뉴스" 탭 추가.
- **이메일 발송**: 앱은 SMTP 클라이언트로 발송. 회사 SMTP는 정책 확인 필요; 불확실 시 Gmail(앱 비밀번호) 또는 SendGrid/메일건 등 발송 전용 서비스 사용 권장.
- **검증·레포팅**: 구현 완료 후 셀프 검증 수행 → 결과를 문서로 레포에 레포팅한다.
